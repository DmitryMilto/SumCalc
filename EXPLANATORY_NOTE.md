# Пояснительная записка

## 1. Назначение работы

В рамках тестового задания было разработано приложение "Калькулятор" на Unity, поддерживающее одну арифметическую операцию - сложение. Приложение сохраняет своё состояние между сеансами, отображает историю вычислений и обрабатывает ошибки пользовательского ввода.

## 2. Что требовалось реализовать

По техническому заданию необходимо было:

- поддержать только операцию сложения;
- выводить `Error`, если выражение не может быть вычислено;
- показывать диалог ошибки;
- после закрытия диалога восстанавливать последнее введённое выражение;
- сохранять текущее выражение и историю вычислений между сеансами;
- следовать `Clean Architecture`;
- использовать шаблон `MVP`;
- разбить решение на независимые assembly-модули.

## 3. Архитектурное решение

Проект разделён на несколько уровней.

### 3.1. Core

Слой `Core` отвечает только за предметную логику вычисления выражения.

Основной файл:

- [ExpressionCalculator.cs](Assets/Scripts/Calculator/Core/ExpressionCalculator.cs)

Реализация намеренно простая: выражение считается корректным только в формате `число+число`. Для этого используется регулярное выражение:

```text
^\d+\+\d+$
```

Такой подход покрывает требования ТЗ и исключает лишнюю сложность парсера.

### 3.2. Application

Слой `Application` содержит use case'ы и описание состояния:

- [CalculateExpressionUseCase.cs](Assets/Scripts/Calculator/Application/CalculateExpressionUseCase.cs)
- [LoadCalculatorStateUseCase.cs](Assets/Scripts/Calculator/Application/LoadCalculatorStateUseCase.cs)
- [SaveCalculatorStateUseCase.cs](Assets/Scripts/Calculator/Application/SaveCalculatorStateUseCase.cs)
- [CalculatorState.cs](Assets/Scripts/Calculator/Application/CalculatorState.cs)

Этот слой ничего не знает о Unity UI и работает только с абстракциями.

### 3.3. Presentation

Слой `Presentation` реализован через `MVP`.

- `View` сообщает о действиях пользователя.
- `Presenter` реагирует на ввод, вызывает use case'ы и обновляет view.

Основные файлы:

- [ICalculatorView.cs](Assets/Scripts/Calculator/Presentation/ICalculatorView.cs)
- [CalculatorPresenter.cs](Assets/Scripts/Calculator/Presentation/CalculatorPresenter.cs)

`CalculatorPresenter`:

- получает изменения ввода;
- выполняет расчёт по нажатию кнопки;
- обновляет текущий результат;
- обновляет историю;
- сохраняет состояние;
- вызывает показ диалога ошибки.

### 3.4. Dialogs

Диалог ошибки вынесен в отдельный модуль:

- [IErrorDialogView.cs](Assets/Scripts/Dialogs/IErrorDialogView.cs)
- [ErrorDialogPresenter.cs](Assets/Scripts/Dialogs/ErrorDialogPresenter.cs)

Это соответствует требованию ТЗ о модульности и позволяет переиспользовать диалог в другом проекте.

### 3.5. Unity layer

Unity-слой содержит только адаптеры:

- [CalculatorViewBehaviour.cs](Assets/Scripts/Unity/Views/CalculatorViewBehaviour.cs)
- [ErrorDialogViewBehaviour.cs](Assets/Scripts/Unity/Views/ErrorDialogViewBehaviour.cs)
- [CalculatorAppBootstrap.cs](Assets/Scripts/Unity/Bootstrap/CalculatorAppBootstrap.cs)

Именно этот слой связывает Unity-сцену, presenter'ы и хранилище состояния.

## 4. Работа со сценой и UI

Интерфейс собран как обычная Unity-сцена:

- [SampleScene.unity](Assets/Scenes/SampleScene.unity)

Сцена не генерируется editor-скриптами и поддерживается вручную. Это упрощает проверку проекта и убирает лишний служебный код.

При реализации UI была поставлена цель визуально приблизить экран к приложению A из ТЗ:

- чёрный фон;
- белая карточка калькулятора;
- строка ввода с подчёркиванием;
- кнопка `RESULT`;
- список истории справа;
- отдельный диалог ошибки.

## 5. Обработка корректных и некорректных выражений

### 5.1. Корректный ввод

Для строки вида `54+21`:

- выполняется расчёт;
- в верхней зоне результата отображается `54+21=75`;
- в историю добавляется запись `54+21=75`;
- состояние сохраняется.

### 5.2. Некорректный ввод

Для строки вида `45+-88` или `98.12+48.1`:

- в результате фиксируется ошибка;
- в историю добавляется запись вида `45+-88=ERROR`;
- показывается диалог с просьбой проверить выражение;
- после закрытия диалога исходный ввод восстанавливается в поле.

Такое поведение было выбрано, потому что оно соответствует приложению A, где ошибочные операции также попадают в историю.

## 6. Сохранение состояния

Состояние приложения состоит из:

- текущего выражения;
- истории вычислений.

По умолчанию используется `PlayerPrefs`, так как это самый простой и надёжный способ для небольшого тестового задания.

Поддерживаются также альтернативные варианты хранения:

- `JsonFile`
- `PlayerPrefsAndJsonFile`

Реализации:

- [PlayerPrefsCalculatorStateRepository.cs](Assets/Scripts/Unity/Infrastructure/PlayerPrefsCalculatorStateRepository.cs)
- [JsonFileCalculatorStateRepository.cs](Assets/Scripts/Unity/Infrastructure/JsonFileCalculatorStateRepository.cs)
- [CompositeCalculatorStateRepository.cs](Assets/Scripts/Unity/Infrastructure/CompositeCalculatorStateRepository.cs)

После повторного запуска приложения:

- восстанавливается последнее выражение;
- восстанавливается история;
- верхняя запись истории показывается как текущий результат.

## 7. Тестирование

Для проверки ключевых сценариев добавлены EditMode-тесты:

- [CalculatorPresenterTests.cs](Assets/Tests/EditMode/CalculatorPresenterTests.cs)

Покрыты следующие случаи:

- корректный расчёт суммы;
- ошибка при некорректном выражении;
- показ диалога ошибки;
- восстановление выражения после закрытия диалога;
- восстановление сохранённого состояния.

## 8. Принятые упрощения

В решении сознательно приняты следующие упрощения:

- поддерживается только выражение из двух операндов и одного символа `+`;
- не поддерживаются пробелы, отрицательные числа, десятичные дроби и цепочки операций;
- не используется сложный математический парсер, так как ТЗ допускает сделать "так, как проще всего".

Эти упрощения не противоречат требованиям задания и позволяют сохранить решение компактным и понятным.

## 9. Соответствие требованиям ТЗ

### Реализовано

- одна арифметическая операция - сложение;
- ошибка при некорректном вводе;
- диалог ошибки;
- восстановление введённого выражения после закрытия диалога;
- сохранение выражения и истории между сеансами;
- MVP;
- разделение на assembly-модули;
- структура, близкая к Clean Architecture.

### Что важно понимать

- Парсер реализован в минимальном необходимом объёме.
- UI не копирует макеты пиксель-в-пиксель, но следует их структуре и пользовательскому сценарию.

## 10. Итог

В результате выполнено модульное Unity-приложение, которое соответствует ключевым требованиям тестового задания:

- решает задачу сложения;
- корректно обрабатывает ошибки;
- сохраняет состояние;
- имеет разделение по слоям и модулям;
- покрыто базовыми тестами.

Решение ориентировано на простоту, читаемость и проверяемость, без избыточной сложности для данного объёма задачи.
